OOP

Object Oriented Programming

Object oriented Programming (OOP) is a paradigm accepted in industry and education that creates an object, which is a value or variable that has associated methods that are functions associated to a particular type.

Below is an example with a method Seconds from standard library.

const day = 24 * time.Hour
fmt.Println(day.Seconds()) // "86400"

Methods Declarations

Methods are declared with an extra parameter that appears before the function name.

type Point struct{ X, Y float64 }

// traditional function
func Distance(p, q Point) float64 {
    return math.Hypot(q.X-p.X, q.Y-p.Y)
}

// same thing, but as a method of the Point type
func (p Point) Distance(q Point) float64 {
    return math.Hypot(q.X-p.X, q.Y-p.Y)
}

func main() {
    p := Point{1, 2}
    q := Point{4, 6}
    fmt.Println(Distance(p, q)) // "5", function call
    fmt.Println(p.Distance(q))  // "5", method call
}

Methods can be associated with any named types defined into the same package as long as the underluing type is neither a pointer or an interface.

Methods with pointer receiver

Calling a function makes a copy of each argument value. To avoid copying large quantities of memory we pass the address of variable using a pointer. (See variables). The same goes for methods, if a method needs to update a pointer receiver we pass the address memory of type as pointer.

func (p *Point) ScaleBy(factor float64) {
     p.X *= factor
     p.Y *= factor
 }

// call (*Point).ScaleBy method 
r := &Point{1, 2}
r.ScaleBy(2)
fmt.Println(*r) // "{2, 4}"

// alternative syntax
p := Point{1, 2}
pptr := &p
pptr.ScaleBy(2)
fmt.Println(p) // "{2, 4}"

// alternative syntax
p := Point{1, 2}
(&p).ScaleBy(2)
fmt.Println(p) // "{2, 4}"

Go allows to avoid the two alternative syntax with:

p.ScaleBy(2)

and the compiler will perform an implicit &p on the variable. But we can not do this:

Point{1, 2}.ScaleBy(2) // compile error: can't take address of Point literal

because we cannot call a *Point method on a temporary value.

nil value

On methods passing nil as value is correct because nil is a meaningful zero data type for slices and maps.
